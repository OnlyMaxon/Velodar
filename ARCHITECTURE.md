# Bike Radar — Architecture

> Onboarding doc for anyone (human or AI agent) opening this repo fresh.
> Read this first, then `README.md` for setup/run commands.

## What this is

**Bike Radar** — "Waze for cyclists" in Poland. A community app where cyclists
report **police posts, speed cameras and e-bike controls**, see them as live map
markers, and vote them up/down (Waze-style "still there / gone"). It also does
**live speed tracking** with a visual alert when you exceed the 25 km/h e-bike
assist limit.

- **Client:** React Native / **Expo bare workflow** with `expo-dev-client`
  (NOT Expo Go — needs native `react-native-maps` + `expo-location`).
  Expo **SDK 52** / React Native **0.76.9** / React 18.3.1.
- **Backend:** **Supabase** — Postgres + **PostGIS** + **Realtime** + **anonymous Auth**.
- **Builds:** locally on device via `expo run:android|ios --device`. No EAS, no
  App Store / TestFlight, no paid Apple Developer account (iOS uses a free
  Personal Team).

Working directory: `D:\VS\WazeVelo` (Windows). Ignore any `/home/claude/...`
paths in old prompts — they were from a prior sandbox and don't exist here.

## Runtime data flow

### Auth
`App.js` calls `ensureSession()` on mount → if no session, `signInAnonymously()`.
Every device gets a stable `auth.uid()` with the **`authenticated`** role (yes,
anonymous users are `authenticated`), which is what RLS and one-vote-per-user
depend on. No signup/login UI.

### Reports — pull + push (`useReports.js`)
- **Pull (authoritative):** `nearby_reports(lat, lng, radius_m)` RPC returns
  active, non-expired reports within radius (PostGIS `ST_DWithin`), pre-joined
  with the caller's own vote (`my_vote`) and `distance_m`. Called on mount, when
  the user drifts > 3 km, and on manual `refresh()`.
- **Push (live):** a Realtime channel on `public.reports` reconciles
  INSERT/UPDATE/DELETE into local state. Realtime rows lack `distance_m`/`my_vote`
  (RPC-computed), so inserts are distance-filtered client-side; the next pull
  backfills `my_vote`.
- **Local prune:** every 30 s, expired rows are dropped client-side as a safety
  net even if the server signal is late.

### Voting (`castVote` → `cast_vote` RPC)
One vote per user per report (`unique(report_id, user_id)` + `ON CONFLICT` to
update). Counts are recomputed from `report_votes` (single source of truth).

### Speed (`useSpeedTracking.js`)
Own high-rate `watchPositionAsync` (BestForNavigation, ~1 Hz). Pipeline:
prefer `coords.speed` → else derive from distance/time → reject poor-accuracy
fixes & GPS teleports → **median of a 5-sample window** → **EMA** → clamp small
values to 0. Over-limit alert uses **hysteresis** (on at 28 km/h, off at 25) so
it doesn't flicker at the limit. `SPEED_LIMIT_KMH = 25`.

### Expiry / "gone" (server-side, in the SQL)
Reports default to ~35 min life (`expires_at`). A `cast_vote('up')` nudges
expiry out (capped at 60 min from creation). Enough down-votes (**≥3 and more
than the ups**) flips `is_active=false` immediately. `expire_reports()` runs via
`pg_cron` every minute to flip expired rows so the change propagates over
Realtime; if `pg_cron` is unavailable, reads still filter by `expires_at` and the
client prunes locally.

## Backend contract (`supabase/migrations/0001_init.sql`)

**Tables**
- `reports(id, type, lat, lng, geog, created_at, expires_at, upvotes, downvotes, is_active, created_by)`
  — `geog geography(Point,4326)` kept in sync from lat/lng by a trigger; GiST index.
- `report_votes(id, report_id, user_id, vote, created_at)` — `unique(report_id, user_id)`.

**Enums:** `report_type` = `police | camera | ebike_control`; `vote_type` = `up | down`.

**RPCs** (all `SECURITY DEFINER`; clients never write tables directly)
| Function | Args | Returns | Notes |
|---|---|---|---|
| `nearby_reports` | `in_lat, in_lng, radius_m=15000` | rows + `distance_m`, `my_vote` | read path |
| `create_report` | `in_type, in_lat, in_lng` | `uuid` | sets `created_by = auth.uid()` |
| `cast_vote` | `in_report_id, in_vote` | void | dedup + recount + gone/extend logic |
| `expire_reports` | — | `int` | maintenance (pg_cron 1/min) |

**RLS:** `reports` SELECT allowed to `authenticated` (`using(true)`); no direct
INSERT/UPDATE policy — writes go only through the `SECURITY DEFINER` RPCs.
`report_votes` SELECT limited to own rows. Reads with just the publishable key
and **no session** hit the `anon` role and are correctly denied — sign in first.

**Realtime:** `public.reports` is in the `supabase_realtime` publication with
`replica identity full`.

## Configuration / env

`.env` (gitignored; template in `.env.example`):
- `EXPO_PUBLIC_SUPABASE_URL` — project URL (ref `ajhkfnyprryornvyvoys`).
- `EXPO_PUBLIC_SUPABASE_ANON_KEY` — the **publishable** key (`sb_publishable_...`,
  the modern public/anon key). Public by design; security is RLS. Never a
  `service_role`/`sb_secret_...` key here.
- `GOOGLE_MAPS_ANDROID_API_KEY` — Android only (iOS uses Apple Maps). Empty for now.

Expo requires the `EXPO_PUBLIC_` prefix to inline vars into the JS bundle (the
user's original `NEXT_PUBLIC_*` names are a Next.js convention and won't work).

## Map provider

`MapScreen` does **not** force `PROVIDER_GOOGLE`. Default provider →
Google Maps on Android (its only option; needs the API key in app config),
Apple Maps on iOS (no key). So iOS shows a working map with zero extra setup;
Android tiles stay blank until `GOOGLE_MAPS_ANDROID_API_KEY` is set **and**
`expo prebuild -p android --clean` (or `run:android`) is re-run to inject the
`com.google.android.geo.API_KEY` meta-data into the manifest.

## Non-obvious decisions (don't "fix" these blindly)

- **Bottom `ReportCard`, not a Marker `Callout`** — Callout button presses are
  unreliable on Android; the card gives dependable voting.
- **`tracksViewChanges`** on markers is on briefly then off (rasterises the emoji
  view on Android without killing pan performance).
- **Two location watchers** on purpose: `useLocation` (cheap, for map/placement)
  vs `useSpeedTracking` (high-rate, navigation-tuned). Don't merge them casually.
- **Speed hysteresis** (28 on / 25 off) is intentional anti-flicker, not a bug.
- **`android/` is generated** — edit `app.config.js`, not native files; regenerate
  with prebuild.

## Gotchas / do-not

- Node 24 works but Expo targets Node **20/22 LTS**; if tooling misbehaves, try Node 22.
- Do **not** run `npm audit fix --force` — the reported vulns are transitive RN
  deps; forcing breaks native versions.
- Align native module versions with `npx expo install --check` / `--fix`, not by
  hand-pinning in `package.json`.
